# Plots — `results/main_table_5t/plots/`

Generated by [scripts/plot_main_table.py](../../../scripts/plot_main_table.py) (headline plots) and [scripts/paired_delta_analysis.py](../../../scripts/paired_delta_analysis.py) (paired comparisons). Re-run either script to refresh after the underlying eval `results.json` files change.

## Conventions

Two different averaging methods appear in these plots — they are **not** interchangeable and the difference is annotated on every figure:

- **Macro mean F1** = unweighted mean over the 5 tasks (each task = 1/5). What `mean_f1` in `results.json` always was. Insensitive to per-task imbalance.
- **Micro mean F1** = prompt-count-weighted mean over all per-prompt F1s. Matches the `val_lb_avg_f1` metric reported during LoRA training in [grad_lora_finetuning/trainer.py](../../../grad_lora_finetuning/trainer.py). What you should compare to training-time validation curves.

The micro and macro means agree to within ~1 F1 in this dataset because the per-task counts are not too imbalanced (test: 12–18 per task, extended_test: 31–61 per task). They diverge more on `extended_test` than on `test`.

**Headline metric for all bar plots = micro.** The macro number is annotated underneath the bar in italics where it differs.

Sample sizes:
- `test` split: 70 prompts (qasper 14, 2wikimqa 12, qasper_e 18, hotpotqa_e 12, 2wikimqa_e 14)
- `extended_test` split: 224 prompts (qasper 32, 2wikimqa 54, qasper_e 46, hotpotqa_e 31, 2wikimqa_e 61). This is a strict superset of `test` adding all prompts in the (6500, 8192] token window that the standard `[4096, 6500]` filter would have dropped.

Conditions, in display order:
- **B0** — plain Llama-3.2-1B-Instruct, no eviction, no fine-tuning. Performance ceiling.
- **B1/cs1024**, **B1/cs2048** — recency baseline (keep most recent N tokens, evict oldest).
- **M1** — LoRA SFT only, no NAMM, full cache. Best LoRA possible.
- **M2/cs1024**, **M2/cs2048** — standalone trained NAMM eviction, no LoRA.
- **M4/cs1024**, **M4/cs2048** — LoRA finetuned on top of frozen NAMM (the joint setup).
- **A4/cs1024_no_namm**, **A4/cs2048_no_namm** — ablation: take the M4 LoRA but evaluate it WITHOUT NAMM (cache_size = 8192 = filter_by_length, no token ever evicted). Tests whether NAMM-time training pushes the LoRA out of distribution for plain inference.

## Headline plots (micro mean F1)

### `mean_f1_test.png`, `mean_f1_extended.png`
One bar per condition, sorted by display order. Y-axis = micro mean F1. The bold number above each bar is the micro F1; the small grey number underneath in parentheses is the macro F1. Footer caption explicitly names which average is used.

### `per_task_test.png`, `per_task_extended.png`
Grouped bar chart with one group per task and one bar per condition inside each group. Each bar = mean F1 over that task's prompts in that split. This is the per-task macro (within a task) — aggregation across tasks is done in the `mean_f1_*.png` plots above.

### `cs_sweep.png`, `cs_sweep_extended.png`
Cache-size pairings for the four families that have both cs1024 and cs2048 versions (B1, M2, M4, A4). Two horizontal reference lines: B0 (plain Llama, full cache) and M1 (LoRA only, no NAMM). Y-axis = micro mean F1.

## Paired delta plots — `paired_delta_<A>_vs_<B>_<split>.png`

Generated by `paired_delta_analysis.py`. For each `(model A, model B, split)` triple, the script intersects the prompts that *both* models scored and runs a paired analysis. Eight comparisons × two splits = **16 plots**:

| Comparison | Question it answers |
|---|---|
| `M4/cs1024` vs `A4/cs1024_no_namm` | Same LoRA, with vs without NAMM at eval time → does the NAMM hurt or help? (cs1024) |
| `M4/cs2048` vs `A4/cs2048_no_namm` | Same as above, cs2048 |
| `M1` vs `A4/cs1024_no_namm` | Both eval no-NAMM, different LoRA training (no NAMM vs NAMM cs1024) → does NAMM-time training put the LoRA OOD? |
| `M1` vs `A4/cs2048_no_namm` | Same, cs2048 LoRA |
| `B1/cs1024` vs `M2/cs1024` | Recency vs learned NAMM at the same cache budget → does the learned policy beat the trivial one? |
| `B1/cs2048` vs `M2/cs2048` | Same, cs2048 |
| `M4/cs1024` vs `M4/cs2048` | M4 family across cache sizes |
| `A4/cs1024_no_namm` vs `A4/cs2048_no_namm` | M4 LoRAs evaluated no-NAMM across cache sizes |

Each plot has three panels:

**Panel 1 — micro mean bars (A, B) with paired Δ + bootstrap 95% CI.** Two bars showing the micro F1 of A and B on the shared prompts, and a title line `Δ = +X.XX [95% CI lo, hi] sig`. Significance flag `*` means the percentile bootstrap CI excludes zero (B = 2000, percentile method, paired by prompt index).

**Panel 2 — sorted per-prompt Δ-F1 waterfall.** Each vertical bar = one prompt's `F1(B) − F1(A)`, sorted left → right from biggest negative to biggest positive. Bar colour = which task the prompt belongs to (legend top-left, see also the inline per-task Δ summary). Reading guide:
- Bars mostly above zero → B is better almost everywhere.
- Bars mostly below zero → A is better.
- Most bars near zero with a few extremes → the two models mostly agree, disagreements concentrated on a small subset.
- Colour clustering reveals task-specific failure modes (e.g. all negatives are red `hotpotqa_e` → A wins specifically there).

**Panel 3 — F1 vs prompt length scatter.** Two series, grey = A, blue = B. X = LongBench prompt length in *words* (from the dataset metadata), Y = F1. Reveals length-OOD failure modes: if blue dots collapse past x ≈ 5000 while grey stays high, you have a length-driven failure of B.

## Re-running

```bash
# After new evals or post-hoc patches:
python scripts/organize_eval_results.py        # refresh results/main_table_5t/all_results.json + READMEs
python scripts/plot_main_table.py              # main_table headline plots
python scripts/paired_delta_analysis.py        # 16 paired comparison plots
```

`organize_eval_results.py` is idempotent and copies/symlinks the latest `ext_*` run from each `eval_results/<model>/` source directory. `paired_delta_analysis.py` reads `per_prompt_f1` from the per-condition `results.json` files (so it requires the patcher to have been run on any pre-2026-04-11 evals — see [scripts/patch_micro_mean_f1.py](../../../scripts/patch_micro_mean_f1.py) which adds `micro_mean_f1`/`n_prompts_total` to existing `results.json` files in place).
